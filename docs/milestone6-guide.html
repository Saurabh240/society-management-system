Milestone 6 — General Ledger & Banking

High-level explanation before you write any code

📅 Week 2  |  ~20 hours total  |  10 issues  |  Builds on M5 CoA foundation

What are we building?

Milestone 6 adds two accounting features:
  • General Ledger — the central record of all financial transactions. Every time money moves (dues collected, vendor paid, expense recorded), it creates a journal entry with balanced debit/credit lines. The ledger is the filtered, searchable view of those entries.
  • Banking — a register of all the HOA's bank accounts (operating account, reserve account, savings). Stores masked account numbers, balances, and whether the account supports check printing.

Why does General Ledger depend on CoA?

Every journal entry line references a Chart of Accounts entry (from M5). For example:
  • Debit: Account 1000 (Cash - Operating Account) → money came in
  • Credit: Account 4000 (HOA Fees) → it was fee income
Without CoA being done first, the journal entry form has no accounts to select from. That's the dependency chain: CoA (M5) → Journal Entries → Ledger View (both M6).

What's the relationship between Journal and Ledger?

These are two separate tables but tightly linked:
  • Journal = the source document. "On Apr 6, $1,200 was collected from HOA fees." One journal entry can have many lines (one per account touched).
  • Ledger = the individual impact per account. When you post a journal entry, the system automatically creates one ledger row per journal line. The ledger is what you query when you want "show me all transactions for the Cash account this month."
Think of it as: Journal is the input → Ledger is the indexed, queryable output.

What is Banking about here?

This is just account registration — not actual bank feeds or reconciliation. The HOA admin manually enters their bank accounts (routing number, account number, account name like "Operating Checking"). The system stores it masked (shows only last 4 digits) and uses it later in M7 when bills are paid from a specific bank account. Banking in M6 = CRUD + list, nothing more.

What's NOT in Milestone 6?

  • Bills payment — that's M7. Banking is just registration here.
  • Balance Sheet / Income Statement reports — M7.
  • Automatic bank reconciliation — out of scope entirely for now.
  • The Overview tab stats (Total Revenue, etc.) — those will be wired up at end of M6 once journal data exists.

Key Accounting Concepts

You need to understand these before implementing

1. Double-Entry Bookkeeping (Critical)

Every journal entry must balance: total debits = total credits. This is not optional — it's a mathematical law of accounting. If someone records $1,200 in HOA fees: 
DEBIT  Cash (Account 1000)        $1,200   ← money came into the bank
CREDIT HOA Fees Income (Account 4000) $1,200   ← it was earned as income
Your backend must validate this before saving. If debits ≠ credits → reject with HTTP 400.

2. Normal Balances (for Ledger calculations)

Each account type has a "normal" side:
Account TypeNormal BalanceIncreases with
ASSETSDebitDebit entries
EXPENSESDebitDebit entries
LIABILITIESCreditCredit entries
EQUITYCreditCredit entries
INCOMECreditCredit entries
This matters for M7 reports (Balance Sheet, Income Statement). For M6 you just store the raw debit/credit amounts — no calculation needed yet.

3. Cash vs Accrual Basis

The General Ledger filter has an "Accounting Basis" dropdown (Cash / Accrual). For M6, just store the basis on each ledger entry and filter by it. You don't need to implement full accrual accounting logic — just pass the filter through to the query. HOA management typically uses cash basis anyway.

4. Journal Entry Lines (JournalLine entity)

A single journal entry has a header (date, association, memo) and multiple lines. Each line has exactly one account, one debit amount (or zero), one credit amount (or zero). A line cannot have both debit AND credit — it's one or the other per line. 
Journal: id=1, date=2026-04-06, associationId=2, memo="April dues"
  Line 1: accountId=5 (Cash),     debit=1200, credit=0
  Line 2: accountId=12 (HOA Fees), debit=0,    credit=1200

5. Bank Account Security

Never return the full account number in API responses. Store the full number in the DB but always mask it in the response: keep only the last 4 digits, prefix with "****". The routing number is less sensitive but you should still validate it's exactly 9 digits before saving.

General Ledger — How It Works

End-to-end flow from user action to database

User journey

1
User navigates to Accounting → General Ledger tab. Sees filter panel (Association, Account, Date Range, Basis) and an empty transaction table with "No transactions found" message.
2
User clicks "+ Record General Journal Entry" button. Navigates to full-page form at /dashboard/accounting/journal-entry/create.
3
User fills: Date, Association (dropdown), Memo. In the Journal Entry Lines table, selects accounts from CoA, enters debit or credit per line. Clicks "+ Add 2 Rows" if needed. Totals auto-update at bottom.
4
User clicks "Record Journal Entry". Frontend validates debits = credits locally first, then POSTs to backend. Backend validates again, saves Journal + JournalLines, auto-creates Ledger rows, returns 201.
5
User navigates back to General Ledger tab. Applies filters. Table now shows the posted entry. Each row shows: Date, Account Name, Description, Debit, Credit.

What happens inside the backend on POST journal entry

POST /api/v1/accounting/journal-entries
1. Validate balance — sum(debit) must equal sum(credit) across all lines. If not → 400 Bad Request.
2. Save Journal header — date, associationId, memo, tenantId (auto from BaseEntity).
3. Save JournalLines — one row per line in the request, each referencing the journal ID.
4. Auto-create Ledger entries — for each JournalLine, insert one row into ledger_entries with the same date, account, debit, credit, plus accountingBasis = CASH (default). This is what the Ledger tab queries.
5. Return — JournalResponse with all saved data including generated IDs.

Why two tables (Journal + Ledger)?

  • Journal = immutable source of truth. Never modified after posting. Grouped by transaction.
  • Ledger = queryable by account. You can ask "all Cash account entries this month" without joining and grouping journal lines every time. The ledger is pre-indexed by account and date for fast filtering.
  • In accounting terms: journal is the book of original entry, ledger is the book of final entry.

General Ledger — Backend Step by Step

Exact order to implement — each step builds on the previous

Step 1 — Create DB migrations (V20)

Create V20__create_journal_tables.sql. Two tables in one migration since they're created together: 
CREATE TABLE journal_entries (
    id             BIGSERIAL PRIMARY KEY,
    tenant_id      BIGINT       NOT NULL,
    association_id BIGINT       NOT NULL,
    date           DATE         NOT NULL,
    memo           TEXT,
    created_at     TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);

CREATE TABLE journal_lines (
    id             BIGSERIAL PRIMARY KEY,
    journal_id     BIGINT         NOT NULL REFERENCES journal_entries(id) ON DELETE CASCADE,
    account_id     BIGINT         NOT NULL,  -- references chart_of_accounts.id
    description    VARCHAR(255),
    debit          NUMERIC(15,2)  NOT NULL DEFAULT 0,
    credit         NUMERIC(15,2)  NOT NULL DEFAULT 0
);

CREATE TABLE ledger_entries (
    id               BIGSERIAL PRIMARY KEY,
    tenant_id        BIGINT        NOT NULL,
    journal_id       BIGINT        NOT NULL,
    account_id       BIGINT        NOT NULL,
    association_id   BIGINT        NOT NULL,
    date             DATE          NOT NULL,
    description      VARCHAR(255),
    debit            NUMERIC(15,2) NOT NULL DEFAULT 0,
    credit           NUMERIC(15,2) NOT NULL DEFAULT 0,
    accounting_basis VARCHAR(10)   NOT NULL DEFAULT 'CASH',
    created_at       TIMESTAMPTZ   NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_journal_tenant      ON journal_entries(tenant_id);
CREATE INDEX idx_ledger_tenant_date  ON ledger_entries(tenant_id, date);
CREATE INDEX idx_ledger_account      ON ledger_entries(tenant_id, account_id);
CREATE INDEX idx_ledger_assoc        ON ledger_entries(tenant_id, association_id);
Why ON DELETE CASCADE on journal_lines? If a journal entry is ever deleted (shouldn't happen in production but useful for dev cleanup), its lines go with it automatically. The ledger entries referencing it should also be cleaned — add a similar cascade there too.

Step 2 — Implement Journal entity + JournalLine entity

Journal.java — extends BaseEntity:
  • Fields: id, associationId, date (LocalDate), memo
  • Has @OneToMany(mappedBy="journal", cascade=ALL, orphanRemoval=true) to JournalLine list
  • Uses @SuperBuilder — same pattern as Association.java
JournalLine.java — does NOT extend BaseEntity (no tenantId needed, it's scoped via journal):
  • Fields: id, accountId, description, debit (BigDecimal), credit (BigDecimal)
  • @ManyToOne @JoinColumn(name="journal_id") back to Journal
Why LocalDate not Instant for journal date? Journal entry dates are calendar dates (April 6, 2026) not timestamps. Users pick a date, not a moment. LocalDate maps to SQL DATE type which is exactly right. Instant would add unnecessary time-of-day complexity and timezone issues.

Step 3 — Implement Ledger entity

Ledger.java — extends BaseEntity:
  • Fields: id, journalId, accountId, associationId, date (LocalDate), description, debit, credit, accountingBasis (String: "CASH" or "ACCRUAL")
  • No relationship back to Journal — just stores journalId as a foreign key value

Step 4 — Implement repositories

JournalRepository — simple JpaRepository, just needs findByTenantId.

LedgerRepository — needs a flexible filter query. Use @Query with optional params: 
@Query("""
    SELECT l FROM Ledger l
    WHERE l.tenantId = :tenantId
      AND (:associationId IS NULL OR l.associationId = :associationId)
      AND (:accountId     IS NULL OR l.accountId     = :accountId)
      AND (:from          IS NULL OR l.date          >= :from)
      AND (:to            IS NULL OR l.date          <= :to)
      AND (:basis         IS NULL OR l.accountingBasis = :basis)
    ORDER BY l.date DESC
    """)
List<Ledger> findFiltered(
    @Param("tenantId")      Long tenantId,
    @Param("associationId") Long associationId,
    @Param("accountId")     Long accountId,
    @Param("from")          LocalDate from,
    @Param("to")            LocalDate to,
    @Param("basis")         String basis
);
Why nullable params in JPQL? This lets you call one query method for all filter combinations instead of writing separate methods for every possible combination of filters. The :param IS NULL OR pattern short-circuits when a param is not provided.

Step 5 — Implement JournalService

The most important service in M6. Key logic in createJournalEntry():
  1. Validate: sum(debit) == sum(credit) across all lines → throw 400 if not
  2. Build and save Journal entity with lines via cascade
  3. For each saved line, create a Ledger entity and save all in one saveAll() call
  4. Inject CoaRepository to look up account names for the response
  5. Return JournalResponse with full detail
Why validate balance in the service not the controller? Business rules belong in the service layer. The controller just maps HTTP → Java and delegates. If the balance check was in the controller, it would be skipped if you called the service from anywhere else (e.g. a scheduler or another service).

Step 6 — Implement LedgerService + Controllers

LedgerService.getLedgerEntries():
  • Takes all filter params (can all be null)
  • Calls LedgerRepository.findFiltered()
  • Joins with CoA to get account names for display (either via join in JPQL or separate lookup)
  • Maps to LedgerEntryResponse
JournalController: POST /api/v1/accounting/journal-entries
LedgerController: GET /api/v1/accounting/ledger?associationId=&accountId=&from=&to=&basis=

Step 7 — Wire Overview stats (end of M6)

Now that ledger entries exist, update AccountingOverviewController to return real values:
  • totalRevenue = sum of credits on INCOME accounts in ledger
  • totalExpenses = sum of debits on EXPENSE accounts in ledger
  • netIncome = totalRevenue - totalExpenses
  • outstanding = leave as zero until M7 bills exist

Banking — How It Works

Simpler than General Ledger — pure CRUD with security considerations

User journey

1
User goes to Accounting → Banking tab. Sees an association filter dropdown and a table with Association, Name, Account Number (masked), Balance columns.
2
User clicks "+ Add Bank Account". Navigates to full-page form at /dashboard/accounting/banking/create.
3
User fills in: Association, Bank Account Name, Account Type (Checking/Savings/Money Market), Country, Routing Number (9 digits), Account Number, Confirm Account Number, Notes. Toggles "Enable Check Printing" if needed.
4
Frontend validates: routing number = exactly 9 digits, account numbers match. POSTs to backend.
5
Backend validates again, stores the account number masked (only last 4 digits in the response), saves to DB, returns 201.
6
User returns to Banking tab. New account appears in the table with "****1234" as the account number.

The masking logic

In BankingService, before saving: 
// Store ONLY the last 4 digits — never the full number
String masked = "****" + accountNumber.substring(accountNumber.length() - 4);
banking.setAccountNumberMasked(masked);
// Do NOT store the full account number in the DB at all for this MVP
Why not store the full number encrypted? For an HOA SaaS MVP, storing only the masked version is the right tradeoff. Full encryption requires key management infrastructure. If the business needs full numbers later (for ACH payments), that's a separate security feature with proper PCI-DSS compliance.

Banking vs CoA — what's the difference?

Chart of AccountsBank Accounts (Banking)
Abstract ledger categories (Asset, Income, etc.)Actual bank accounts at real financial institutions
Used for double-entry bookkeeping classificationUsed to know which physical account holds money
Example: "1000 - Cash - Operating Account"Example: "Chase Checking ****5678, balance $12,450"
No routing/account numbersHas routing number, masked account number, balance
In M7, when a bill is paid, you'll select both: which expense account (CoA) and which bank account (Banking) the payment comes from.
