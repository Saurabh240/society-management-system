Bills — What It Is

The business concept

An HOA regularly pays vendors for services — a landscaping company mows the common areas, a plumber fixes a broken pipe, an electrician services the parking lot lights. Each of these is a Bill: money the HOA owes to a vendor. Bills are the opposite of invoices — an invoice is money owed TO the HOA, a bill is money owed BY the HOA.

The bills module tracks:
  • Which vendor is owed money and for what association
  • When the bill was issued and when it's due
  • What services/items the bill covers (line items, each linked to a CoA expense account)
  • Whether it's UNPAID, PAID, or OVERDUE

Bill number auto-generation (BILL-001 pattern)

Each bill gets a sequential number per tenant: BILL-001, BILL-002, etc. The admin can override it by entering their own bill number, otherwise it's auto-generated. In the service, auto-generation works by counting existing bills for the tenant and padding: 
long count = billRepository.countByTenantId(tenantId);
String billNumber = "BILL-" + String.format("%03d", count + 1);
Why not a DB sequence? A DB sequence never resets and is shared across all tenants. The BILL-001 pattern is per-tenant, so it needs to be computed per tenant. The count approach is simpler for the MVP scope.

Bill status lifecycle

StatusWhen setUI badge colorPay button shown
UNPAIDDefault on creationYellowYes
PAIDWhen admin clicks PayGreenNo
OVERDUEWhen dueDate < today and still UNPAID — set by scheduled jobRedYes
OVERDUE transition: A @Scheduled(fixedDelay = 3_600_000) job runs hourly in BillService. It finds all UNPAID bills where dueDate < today and bulk-updates their status to OVERDUE. No manual trigger needed.

The Pay Bill flow — what happens inside

When an admin pays a bill, three things happen in one transaction:
  1. The bill's status changes to PAID and paidAt is set to now
  2. The paidFromBankAccountId is recorded on the bill
  3. A balanced journal entry is auto-created in the ledger: 
    DEBIT  Accounts Payable (or the expense account)  $1,200
CREDIT Cash / Bank Account                          $1,200
This is the same JournalService.createJournalEntry() from M6 — Bills reuse that exact service method. The accounting record is fully automated.
Which accounts to use for the journal entry? For simplicity: debit each BillLineItem's expenseAccountId (since the expense was already recorded when the bill was created), credit the bank account's linked CoA account (account 1000 - Cash). For a proper AP workflow you'd debit Accounts Payable — that's a future enhancement. The MVP approach is: debit expense accounts from the bill lines, credit cash.

Step 2 — BillStatus enum + Bill entity + BillLineItem entity

BillStatus.java: Simple enum — UNPAID, PAID, OVERDUE

Bill.java — extends BaseEntity. Fields: id, billNumber, vendorId, associationId, issueDate (LocalDate), dueDate (LocalDate), status (BillStatus enum), totalAmount (BigDecimal), memo, paidAt (Instant), paidFromBankAccountId. Has @OneToMany to BillLineItem with cascade ALL.

BillLineItem.java — does NOT extend BaseEntity. Fields: id, bill (ManyToOne FK), description, expenseAccountId, amount.
Why doesn't BillLineItem extend BaseEntity? Line items are child records scoped entirely through their parent Bill. Tenant isolation is guaranteed by the Bill's tenantId. Adding a separate tenantId column on each line item would be redundant and waste storage.

Step 4 — BillService

Key methods:
  • createBill(request): Auto-generate billNumber if blank → build Bill + lines (cascade) → compute totalAmount = sum(line amounts) → save → return response
  • listBills(associationId, status, from, to): Filter query → join vendor + association names for response

  • payBill(id, request): See the full flow below

payBill() in detail — the most important method

@Transactional
public BillResponse payBill(Long id, PayBillRequest request) {
    // 1. Fetch and validate
    Bill bill = findForTenant(id);
    if (bill.getStatus() == BillStatus.PAID) {
        throw new IllegalStateException("Bill is already paid");
    }

    // 2. Mark as paid
    bill.setStatus(BillStatus.PAID);
    bill.setPaidAt(Instant.now());
    bill.setPaidFromBankAccountId(request.bankAccountId());
    billRepository.save(bill);

    // 3. Auto-create journal entry — debit expenses, credit cash
    List<JournalLineRequest> journalLines = new ArrayList<>();

    // One debit line per bill line item (expense accounts)
    for (BillLineItem line : bill.getLineItems()) {
        journalLines.add(new JournalLineRequest(
            line.getExpenseAccountId(),
            "Bill payment: " + bill.getBillNumber(),
            line.getAmount(),      // debit
            BigDecimal.ZERO        // credit
        ));
    }

    // One credit line for the cash/bank account (account linked to bank)
    // Use account 1000 (Cash) or look up the bank account's linked CoA account
    journalLines.add(new JournalLineRequest(
        CASH_ACCOUNT_ID,          // configured constant or looked up from CoA
        "Payment from: " + bill.getBillNumber(),
        BigDecimal.ZERO,           // debit
        bill.getTotalAmount()      // credit
    ));

    journalService.createJournalEntry(new JournalRequest(
        bill.getAssociationId(),
        request.paymentDate(),
        "Bill payment: " + bill.getBillNumber(),
        journalLines
    ));

    return toResponse(bill);
}
Cash Account ID: The simplest approach is to query CoA for the account with accountCode "1000" (Cash - Operating Account) and use its ID. Or make it a configurable property. Don't hardcode a number — look it up from the CoA table at runtime.

Reports — What It Is

The two implemented reports

Balance Sheet

A snapshot of the HOA's financial position at a single point in time (the "As of Date"). It shows:
  • Assets — what the HOA owns (cash, receivables)
  • Liabilities — what it owes (payables, loans)
  • Equity — retained earnings, current year earnings
The fundamental equation must hold: Assets = Liabilities + Equity

How it's calculated: For each account in CoA, sum all ledger entry debits and credits up to the asOfDate. Apply normal balance (Assets/Expenses: debit-normal; Liabilities/Equity/Income: credit-normal) to compute the net balance per account. Group by account type.

Income Statement (P&L)

A summary of revenues and expenses over a date range (From → To). Shows whether the HOA made or lost money in that period.
  • Revenue — INCOME accounts (HOA Fees, Late Fees, etc.)
  • Expenses — EXPENSES accounts (Landscaping, Maintenance, etc.)
  • Net Income = Revenue - Expenses (green if positive, red if negative)
How it's calculated: Query ledger entries between from and to dates. Sum credits on INCOME accounts = revenue. Sum debits on EXPENSES accounts = expenses. Net income = revenue - expenses.

Step 3 — Add queries to LedgerRepository

The reports need per-account balance queries. Add to LedgerRepository: 
// Sum ledger balance up to a date, grouped by accountId
// Returns list of [accountId, netBalance] pairs
@Query("""
    SELECT l.accountId,
           SUM(l.debit) - SUM(l.credit) AS balance
    FROM Ledger l
    WHERE l.tenantId = :tenantId
      AND (:associationId IS NULL OR l.associationId = :associationId)
      AND l.date <= :asOfDate
    GROUP BY l.accountId
    """)
List<Object[]> sumBalanceByAccountUpToDate(
    @Param("tenantId")      Long tenantId,
    @Param("associationId") Long associationId,
    @Param("asOfDate")      LocalDate asOfDate
);

// Sum over a date range, grouped by accountId
@Query("""
    SELECT l.accountId,
           SUM(l.debit)  AS totalDebit,
           SUM(l.credit) AS totalCredit
    FROM Ledger l
    WHERE l.tenantId = :tenantId
      AND (:associationId IS NULL OR l.associationId = :associationId)
      AND l.date BETWEEN :from AND :to
    GROUP BY l.accountId
    """)
List<Object[]> sumDebitCreditByAccountInRange(
    @Param("tenantId")      Long tenantId,
    @Param("associationId") Long associationId,
    @Param("from")          LocalDate from,
    @Param("to")            LocalDate to
);

Step 4 — ReportsService

getBalanceSheet() logic:
  1. Call sumBalanceByAccountUpToDate() → get raw balance per accountId
  2. Batch-load all CoA entries for those accountIds
  3. For each CoA account, compute normal balance: Assets/Expenses use debit-normal (debit-credit = positive = asset); Liabilities/Equity/Income use credit-normal (credit-debit = positive = liability)
  4. Group into assets[], liabilities[], equity[] lists
  5. Compute totals, return BalanceSheetResponse
getIncomeStatement() logic:
  1. Call sumDebitCreditByAccountInRange() → get debit+credit per accountId
  2. Load CoA for those accounts
  3. For INCOME accounts: amount = credit - debit (credit-normal)
  4. For EXPENSES accounts: amount = debit - credit (debit-normal)
  5. Build revenue[] and expenses[] lists, compute netIncome
Normal balance reminder: For Income Statement you only want INCOME and EXPENSES type accounts. Filter CoA by accountType IN ('INCOME','EXPENSES') before processing.
