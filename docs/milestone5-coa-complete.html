Milestone 5 — Chart of Accounts

Complete explanation of every file that changes and why, followed by full code

What is being built

Milestone 5 adds the foundation of the accounting module:
  • Overview tab — 4 summary stat cards (Total Revenue, Total Expenses, Net Income, Outstanding)
  • Chart of Accounts — the master list of all ledger accounts (Assets, Liabilities, Equity, Income, Expenses) with search and filter
  • Add/Edit Account — a full-page form to create or update a CoA entry
  • Accounting page layout — the 5-tab shell (Overview, General Ledger, Banking, Bills, Reports) that Milestones 6 and 7 will fill in

Backend — what needs to change

Every backend accounting class is currently an empty stub. The following files need to be implemented from scratch:
  • V19__create_chart_of_accounts.sql — Flyway migration creating the chart_of_accounts table. This runs automatically on startup. Latest migration in your project is V18, so this is V19.
  • AccountType.java — New enum: ASSETS, LIABILITIES, EQUITY, INCOME, EXPENSES. Stored as a VARCHAR in the DB.
  • Coa.java — Entity extending BaseEntity (which auto-sets tenantId and createdAt via @PrePersist). Fields: id, accountCode, accountName, accountType, notes.
  • CoaRepository.java — Converted from empty class to a proper interface extends JpaRepository. Adds tenant-scoped query methods for listing and searching.
  • CoaRequest.java — New DTO record for create/update payload (accountName, accountType, accountCode, notes).
  • CoaResponse.java — New DTO record for API response (id + all fields + createdAt).
  • CoaService.java — Converted from empty class to a @Service. Implements list (with optional search/filter), create, update, delete — all scoped to current tenant via TenantContext.
  • CoaController.java — Converted from empty class to a @RestController. Exposes GET/POST/PUT/DELETE endpoints under /api/v1/accounting/coa. Follows the same ApiResponse.success() wrapper pattern as AssociationController.
  • AccountingOverviewController.java — New controller at /api/v1/accounting/overview returning the 4 stat card values. For now these are calculated from journal/bill data — but since that data doesn't exist yet in M5, it returns zeros with the real calculation wired in for M6/M7.

Frontend — what needs to change

The accounting frontend folder currently has two empty files. Everything is new:
  • accountingApi.js — New API layer file in src/modules/accounting/. All HTTP calls for accounting go here. Uses the existing httpClient which already handles JWT tokens, tenant headers, and refresh.
  • routes.jsx — New routing file for accounting, following the exact same pattern as communicationRoutes. Exports accountingRoutes.
  • App.jsx — One-line change: import and register accountingRoutes inside the dashboard route, exactly like communicationRoutes.
  • AccountingPage.jsx — The tab shell. Uses Outlet for nested routes, same pattern as CommunicationPage.jsx. 5 tabs: Overview, General Ledger, Banking, Bills, Reports.
  • OverviewTab.jsx — Fetches stats from /api/v1/accounting/overview and renders 4 stat cards.
  • ChartOfAccountsPage.jsx — Full-page list (not a tab) with search input, account type filter, table of accounts, 3-dot action menu per row with Edit and Delete, and "+ Add Account" button.
  • AddAccountPage.jsx — Full-page form for create and edit. Used for both — detects edit mode from URL param :id.

Files NOT changing

  • httpClient.js — already handles auth, no changes needed
  • SecurityConfig.java — /api/v1/** is already secured by .anyRequest().authenticated(), no new rules needed
  • BaseEntity.java — used as-is, auto-sets tenantId
  • Dashboard.jsx, Sidebar.jsx — sidebar already shows Accounting nav items from screenshots, no changes needed

Request Flow

How a "List Chart of Accounts" request flows end-to-end

GET /api/v1/accounting/coa?search=cash&type=ASSETS

1
Frontend: User types "cash" in the search box on ChartOfAccountsPage. After 400ms debounce, calls getCoaList("cash", "ASSETS") from accountingApi.js
2
httpClient: Attaches Authorization: Bearer <token> header automatically via request interceptor
3
TenantFilter.java: Extracts tenantId from JWT claims, calls TenantContext.set(tenantId)
4
JwtFilter.java: Validates JWT, sets SecurityContext authentication
5
CoaController.java: Receives request, reads search and type query params, delegates to CoaService.listAccounts(search, type)
6
CoaService.java: Calls TenantContext.get() to get tenantId (already set in step 3). Queries CoaRepository.findByTenantIdAndAccountNameContainingIgnoreCaseAndAccountType(tenantId, "cash", ASSETS)
7
CoaRepository: Spring Data JPA generates the SQL: SELECT * FROM chart_of_accounts WHERE tenant_id = 1 AND LOWER(account_name) LIKE '%cash%' AND account_type = 'ASSETS'
8
CoaService: Maps each Coa entity → CoaResponse record and returns the list
9
CoaController: Wraps in ApiResponse.success(list) and returns ResponseEntity.ok(...)
10
Frontend: res.data.data (ApiResponse wrapper) is set into accounts state → table re-renders with filtered results

Create Account flow

POST /api/v1/accounting/coa

1
User fills form on AddAccountPage and clicks "Save Account"
2
createAccount({ accountName, accountType, accountCode, notes }) called from accountingApi.js
3
CoaController receives POST body as CoaRequest, passes to CoaService
4
CoaService builds Coa entity using Coa.builder(). Does NOT set tenantId — BaseEntity.onPrePersist() auto-sets it from TenantContext
5
coaRepository.save(coa) persists to DB, returns saved entity with generated id
6
CoaService maps to CoaResponse, CoaController returns ResponseEntity.status(201).body(ApiResponse.success(response))
7
Frontend shows toast "Account created", navigates back to Chart of Accounts list

Why V19? Your latest migration is V18__add_template_fields.sql. Flyway runs migrations in version order, so the next one must be V19. The double-underscore naming (V19__) is required by Flyway's default naming convention.

CREATE TABLE chart_of_accounts (
    id            BIGSERIAL PRIMARY KEY,
    tenant_id     BIGINT        NOT NULL,
    account_code  VARCHAR(20),
    account_name  VARCHAR(255)  NOT NULL,

CREATE INDEX idx_coa_tenant_name    ON chart_of_accounts(tenant_id, account_name);

Coa.java — Entity

Replaces the empty stub. Extends BaseEntity for automatic tenantId + createdAt.

Why extend BaseEntity? BaseEntity has a @PrePersist hook that automatically sets tenantId = TenantContext.get() before every INSERT. This means CoaService never needs to manually set tenantId — it's always handled. Missing this would cause cross-tenant data leakage.
Why @SuperBuilder? BaseEntity uses @SuperBuilder so any entity extending it must also use @SuperBuilder (not @Builder) to inherit the builder chain. Using @Builder on Coa would break the builder pattern with BaseEntity fields.
backend/src/main/java/com/gstech/saas/accounting/coa/model/Coa.java REPLACE STUB backend 
package com.gstech.saas.accounting.coa.model;

import com.gstech.saas.platform.common.BaseEntity;
import jakarta.persistence.*;
import lombok.*;
import lombok.experimental.SuperBuilder;

@Entity
@Table(
    name = "chart_of_accounts",
    indexes = {
        @Index(name = "idx_coa_tenant_id",   columnList = "tenant_id"),
        @Index(name = "idx_coa_tenant_type", columnList = "tenant_id, account_type")
    }
)
@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor
@SuperBuilder
public class Coa extends BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(length = 20)
    private String accountCode;

    @Column(nullable = false)
    private String accountName;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    private AccountType accountType;

    @Column(columnDefinition = "TEXT")
    private String notes;
}

CoaRepository.java

Converted from empty class to JpaRepository interface.

Why interface not class? Spring Data JPA repositories must be interfaces extending JpaRepository. Spring generates the implementation at runtime. Your existing stub was a plain class — that needs to change to an interface with the correct signature.
backend/src/main/java/com/gstech/saas/accounting/coa/repository/CoaRepository.java REPLACE STUB backend 
package com.gstech.saas.accounting.coa.repository;

import com.gstech.saas.accounting.coa.model.AccountType;
import com.gstech.saas.accounting.coa.model.Coa;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Repository;

import java.util.List;
import java.util.Optional;

@Repository
public interface CoaRepository extends JpaRepository<Coa, Long> {

    // All accounts for a tenant (no filter)
    List<Coa> findByTenantIdOrderByAccountCodeAsc(Long tenantId);

    // Filter by type only
    List<Coa> findByTenantIdAndAccountTypeOrderByAccountCodeAsc(
            Long tenantId, AccountType accountType);

    // Search by name only (case-insensitive)
    List<Coa> findByTenantIdAndAccountNameContainingIgnoreCaseOrderByAccountCodeAsc(
            Long tenantId, String search);

    // Search by name AND filter by type
    List<Coa> findByTenantIdAndAccountNameContainingIgnoreCaseAndAccountTypeOrderByAccountCodeAsc(
            Long tenantId, String search, AccountType accountType);

    // Tenant-scoped single fetch (for update/delete security check)
    Optional<Coa> findByIdAndTenantId(Long id, Long tenantId);
}

CoaService.java

Converted from empty class to full @Service implementation.

Why check tenantId on update/delete? The findByIdAndTenantId query ensures a tenant can never edit or delete another tenant's accounts — even if they guess the ID. This is the same pattern used in AssociationService.
backend/src/main/java/com/gstech/saas/accounting/coa/service/CoaService.java REPLACE STUB backend 
package com.gstech.saas.accounting.coa.service;

import com.gstech.saas.accounting.coa.dto.CoaRequest;
import com.gstech.saas.accounting.coa.dto.CoaResponse;
import com.gstech.saas.accounting.coa.model.AccountType;
import com.gstech.saas.accounting.coa.model.Coa;
import com.gstech.saas.accounting.coa.repository.CoaRepository;
import com.gstech.saas.platform.tenant.multitenancy.TenantContext;
import jakarta.persistence.EntityNotFoundException;
import jakarta.transaction.Transactional;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;

import java.util.List;

@Slf4j
@Service
@RequiredArgsConstructor
public class CoaService {

    private final CoaRepository coaRepository;

    // ── LIST ──────────────────────────────────────────────────────────────────

    public List<CoaResponse> listAccounts(String search, AccountType type) {
        Long tenantId = TenantContext.get();
        List<Coa> results;

        boolean hasSearch = search != null && !search.isBlank();
        boolean hasType   = type != null;

        if (hasSearch && hasType) {
            results = coaRepository
                .findByTenantIdAndAccountNameContainingIgnoreCaseAndAccountTypeOrderByAccountCodeAsc(
                    tenantId, search.trim(), type);
        } else if (hasSearch) {
            results = coaRepository
                .findByTenantIdAndAccountNameContainingIgnoreCaseOrderByAccountCodeAsc(
                    tenantId, search.trim());
        } else if (hasType) {
            results = coaRepository
                .findByTenantIdAndAccountTypeOrderByAccountCodeAsc(tenantId, type);
        } else {
            results = coaRepository.findByTenantIdOrderByAccountCodeAsc(tenantId);
        }

        return results.stream().map(this::toResponse).toList();
    }

    // ── CREATE ────────────────────────────────────────────────────────────────

    @Transactional
    public CoaResponse createAccount(CoaRequest request) {
        // BaseEntity.onPrePersist() will auto-set tenantId from TenantContext
        Coa coa = Coa.builder()
                .accountName(request.accountName())
                .accountType(request.accountType())
                .accountCode(request.accountCode())
                .notes(request.notes())
                .build();

        Coa saved = coaRepository.save(coa);
        log.info("[CoA] Created account id={} tenantId={}", saved.getId(), saved.getTenantId());
        return toResponse(saved);
    }

    // ── UPDATE ────────────────────────────────────────────────────────────────

    @Transactional
    public CoaResponse updateAccount(Long id, CoaRequest request) {
        Coa coa = findForTenant(id);

        coa.setAccountName(request.accountName());
        coa.setAccountType(request.accountType());
        coa.setAccountCode(request.accountCode());
        coa.setNotes(request.notes());

        Coa saved = coaRepository.save(coa);
        log.info("[CoA] Updated account id={}", id);
        return toResponse(saved);
    }

    // ── DELETE ────────────────────────────────────────────────────────────────

    @Transactional
    public void deleteAccount(Long id) {
        Coa coa = findForTenant(id);
        coaRepository.delete(coa);
        log.info("[CoA] Deleted account id={}", id);
    }

    // ── HELPERS ───────────────────────────────────────────────────────────────

    private Coa findForTenant(Long id) {
        return coaRepository.findByIdAndTenantId(id, TenantContext.get())
                .orElseThrow(() -> new EntityNotFoundException(
                        "Account not found with id=" + id));
    }

    private CoaResponse toResponse(Coa coa) {
        return new CoaResponse(
                coa.getId(),
                coa.getAccountCode(),
                coa.getAccountName(),
                coa.getAccountType(),
                coa.getNotes(),
                coa.getCreatedAt()
        );
    }
}

CoaController.java

Converted from empty class to @RestController. Follows AssociationController pattern exactly.

Why ApiResponse.success() wrapper? Every API in your project returns { success: true, data: ... }. The frontend reads res.data.data to get the payload. Skipping the wrapper would break the frontend pattern.

package com.gstech.saas.accounting.coa.controller;

import com.gstech.saas.accounting.coa.dto.CoaRequest;
import com.gstech.saas.accounting.coa.dto.CoaResponse;
import com.gstech.saas.accounting.coa.model.AccountType;
import com.gstech.saas.accounting.coa.service.CoaService;
import com.gstech.saas.platform.common.ApiResponse;
import io.swagger.v3.oas.annotations.Operation;
import io.swagger.v3.oas.annotations.tags.Tag;
import jakarta.validation.Valid;
import lombok.RequiredArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

import java.util.List;

@RestController
@RequestMapping("/api/v1/accounting/coa")
@RequiredArgsConstructor
@Tag(name = "Chart of Accounts", description = "CoA management APIs")
public class CoaController {

    private final CoaService coaService;

    @Operation(summary = "List all accounts with optional search and type filter")
    @GetMapping
    public ResponseEntity<ApiResponse<List<CoaResponse>>> listAccounts(
            @RequestParam(required = false) String search,
            @RequestParam(required = false) AccountType type) {

        return ResponseEntity.ok(
                ApiResponse.success(coaService.listAccounts(search, type)));
    }

    @Operation(summary = "Create a new account")
    @PostMapping
    public ResponseEntity<ApiResponse<CoaResponse>> createAccount(
            @Valid @RequestBody CoaRequest request) {

        return ResponseEntity
                .status(HttpStatus.CREATED)
                .body(ApiResponse.success(coaService.createAccount(request)));
    }

    @Operation(summary = "Update an existing account")
    @PutMapping("/{id}")
    public ResponseEntity<ApiResponse<CoaResponse>> updateAccount(
            @PathVariable Long id,
            @Valid @RequestBody CoaRequest request) {

        return ResponseEntity.ok(
                ApiResponse.success(coaService.updateAccount(id, request)));
    }

    @Operation(summary = "Delete an account")
    @DeleteMapping("/{id}")
    public ResponseEntity<ApiResponse<Void>> deleteAccount(@PathVariable Long id) {
        coaService.deleteAccount(id);
        return ResponseEntity.ok(ApiResponse.success(null));
    }
}

  const ref = useRef(null);

  useEffect(() => {
    const handler = (e) => { if (ref.current && !ref.current.contains(e.target)) setOpen(false); };
    document.addEventListener("mousedown", handler);
    return () => document.removeEventListener("mousedown", handler);
  }, []);

  return (
    <div className="relative" ref={ref}>
      <button onClick={() => setOpen((p) => !p)} className="p-1 rounded hover:bg-gray-100">

AddAccountPage.jsx

Create and edit form — same page, detects edit mode from URL :id param

Why use route state for edit? When the user clicks Edit from the list, the account data is already in memory. Passing it via navigate(url, { state: { account } }) avoids an extra API call. If the user navigates directly to the edit URL (e.g. bookmark), the page falls back to fetching from the API using the :id param.

const ACCOUNT_TYPES = [
  { value: "ASSETS",      label: "Assets"      },
  { value: "LIABILITIES", label: "Liabilities" },
  { value: "EQUITY",      label: "Equity"      },

];

const labelCls  = "block text-sm font-medium text-gray-700 mb-1";

export default function AddAccountPage() {
  const navigate      = useNavigate();
  const { id }        = useParams();
  const { state }     = useLocation();
  const isEdit        = !!id;

  const [form, setForm]         = useState({
    accountName: "",
    accountType: "ASSETS",
    accountCode: "",
    notes: "",
  });
  const [submitting, setSubmitting] = useState(false);

  // Pre-fill from route state (fast path) or fetch if direct URL
  useEffect(() => {
    if (isEdit) {
      const acct = state?.account;
      if (acct) {
        setForm({
          accountName: acct.accountName || "",
          accountType: acct.accountType || "ASSETS",
          accountCode: acct.accountCode || "",
          notes:       acct.notes       || "",
        });
      }
      // If no state (direct URL navigation), we'd fetch by id here
      // For now the list always passes state — extend if needed
    }
  }, [id, state, isEdit]);

  const set = (key) => (e) => setForm((p) => ({ ...p, [key]: e.target.value }));

  const handleSubmit = async (e) => {
    e.preventDefault();
    if (!form.accountName.trim()) return toast.error("Account name is required");
    if (!form.accountType)        return toast.error("Account type is required");

    setSubmitting(true);
    try {
      const payload = {
        accountName: form.accountName.trim(),
        accountType: form.accountType,
        accountCode: form.accountCode.trim() || null,
        notes:       form.notes.trim()       || null,
      };

      if (isEdit) {
        await updateAccount(id, payload);
        toast.success("Account updated successfully");
      } else {
        await createAccount(payload);
        toast.success("Account created successfully");
      }

      navigate("/dashboard/accounting/chart-of-accounts");
    } catch (err) {
      toast.error(err.response?.data?.error || "Failed to save account");
    } finally {
      setSubmitting(false);
    }
  };

  return (
    <div className="max-w-xl mx-auto py-6 px-4">
      {/* Back button */}
      <button
        onClick={() => navigate("/dashboard/accounting/chart-of-accounts")}
        className="flex items-center gap-1 text-sm text-gray-500 hover:text-gray-700 mb-4 transition group"
      >
        <ChevronLeft size={16} className="group-hover:-translate-x-1 transition-transform" />
        Back to Chart of Accounts
      </button>

      <h1 className="text-2xl font-semibold text-gray-900 mb-6">
        {isEdit ? "Edit Account" : "Add Account"}
      </h1>

      <div className="bg-white border border-gray-200 rounded-xl p-6 shadow-sm">
        <form onSubmit={handleSubmit} className="space-y-5">

          <div>
            <label className={labelCls}>
              Account Name <span className="text-red-500">*</span>
            </label>
            <input
              type="text"
              value={form.accountName}
              onChange={set("accountName")}
              placeholder="Enter account name"
              required
              className={inputCls}
            />
          </div>

          <div>
            <label className={labelCls}>
              Account Type <span className="text-red-500">*</span>
            </label>
            <select
              value={form.accountType}
              onChange={set("accountType")}
              required
              className={inputCls}
            >
              {ACCOUNT_TYPES.map((t) => (
                <option key={t.value} value={t.value}>{t.label}</option>
              ))}
            </select>
          </div>

          <div>
            <label className={labelCls}>Account Number <span className="text-gray-400">(optional)</span></label>
            <input
              type="text"
              value={form.accountCode}
              onChange={set("accountCode")}
              placeholder="Enter account number (optional)"
              className={inputCls}
            />
          </div>

          <div>
            <label className={labelCls}>Notes</label>
            <textarea
              value={form.notes}
              onChange={set("notes")}
              placeholder="Enter notes about this account"
              rows={4}
              className={`${inputCls} resize-y`}
            />
          </div>

          <div className="flex gap-3 pt-2">
            <button
              type="submit"
              disabled={submitting}
              className="flex items-center gap-2 px-5 py-2 text-sm text-white rounded-lg hover:opacity-90 transition disabled:opacity-50"
              style={{ backgroundColor: "var(--color-primary)" }}
            >
              {submitting && <Loader2 size={14} className="animate-spin" />}
              {isEdit ? "Update Account" : "Save Account"}
            </button>
            <button

              className="px-5 py-2 text-sm border border-gray-300 rounded-lg text-gray-600 hover:bg-gray-50 transition"
            >
              Cancel
            </button>
          </div>

        </form>
      </div>
    </div>
  );
